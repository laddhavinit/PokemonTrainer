🎮 Pokémon Trainer App - Complete Architecture Guide

📋 Project Overview

A modern Angular application that allows users to browse and collect Pokémon using the PokéAPI. Built with Angular 18+ features including standalone components, signals, route guards, and advanced pagination. Features a clean Pokéball-inspired design with persistent user data and real-time state management.

🎯 Core Features

  • User Authentication: Trainer name-based login with route protection
  • Pokémon Catalogue: Paginated browsing of 200 Pokémon (20 per page)
  • Personal Collection: View and manage caught Pokémon with real-time updates
  • Persistent Data: User-specific data storage that survives logout/login cycles
  • Route Guards: Prevent unauthorized access and auto-redirect logged-in users
  • Responsive Design: Mobile-first approach with 4-card grid layout
  • Smart Pagination: Efficient navigation with page indicators and smooth scrolling

🚦 Route Configuration & Guards

Route Protection Strategy

export const routes: Routes = [ { path: '', component: LandingComponent, canActivate: [LandingGuard] // Redirects if already logged in }, { path: 'catalogue', component: CatalogueComponent, canActivate: [AuthGuard] // Requires authentication }, { path: 'trainer', component: TrainerComponent, canActivate: [AuthGuard] // Requires authentication } ];

✅ Guard Logic

  • AuthGuard: Checks if trainer name exists in localStorage
  • LandingGuard: Redirects logged-in users to catalogue
  • SSR Safe: Platform checks prevent server-side errors

💾 Data Persistence Strategy

🔑 User-Specific Storage

Each trainer's data is stored separately using their name as a key prefix.

// Storage Keys trainerName: "Ash" caughtPokemon_Ash: [1, 4, 7, 25] caughtPokemon_Misty: [7, 54, 120] allPokemon: [...] // Shared cache

🔄 Data Lifecycle

  1. Login: Store trainer name
  2. First Load: Fetch from API, cache in sessionStorage
  3. Subsequent Loads: Read from sessionStorage
  4. Catch/Release: Update localStorage with trainer-specific key
  5. Logout: Clear trainer name, preserve Pokémon data
  6. Re-login: Restore previous collection

🔄 State Management Flow

Signal-Based Reactivity

The app uses Angular Signals for reactive state management:

// 1. Base Signals (Sources of Truth) private pokemonListSignal = signal<Pokemon[]>([]); private caughtIdsSignal = signal<number[]>([]); // 2. Computed Signals (Derived State) pokemonList = computed(() => { const allPokemon = this.pokemonListSignal(); const caughtIds = this.caughtIdsSignal(); return allPokemon.map(p => ({ ...p, caught: caughtIds.includes(p.id) })); }); caughtPokemon = computed(() => this.pokemonList().filter(p => p.caught) ); // 3. Component Usage (Automatic Updates) get pokemonList() { return this.pokemonService.pokemonList(); // Auto-updates UI }

✅ Benefits of This Architecture

  • Automatic Updates: UI updates when signals change
  • Performance: Only re-renders when necessary
  • Type Safety: Full TypeScript support
  • Debugging: Clear data flow and state tracking
